## 镜像的本质

我们在其他场合也见过"镜像"这个词：光盘镜像、重装电脑用的硬盘镜像、虚拟机系统镜像。它们有共同点：只读、不允许修改、以标准格式存储一系列文件，需要时再从中提取数据运行起来。

容器技术里的镜像同理。容器由操作系统动态创建，可以用一种办法把它的初始环境固化下来，保存成静态文件，相当于把容器"拍扁"，方便存放、传输、版本化管理。

## 镜像与容器的比喻

延续第 02 讲"小板房"的比喻：

- **镜像 = 样板间**：把运行进程所需的文件系统、依赖库、环境变量、启动参数等所有信息打包整合到一起
- 镜像文件无论放在哪里，操作系统都能根据这个"样板间"快速重建容器，应用看到的会是一致的环境

## 镜像与 rpm/deb 安装包的区别

从功能上看，镜像和常见的 tar、rpm、deb 等安装包一样都打包了应用程序，但最大不同在于：镜像不仅有基本可执行文件，还有应用运行时的整个系统环境。

| 维度 | tar/rpm/deb 安装包 | 容器镜像 |
|------|-------------------|----------|
| 打包内容 | 仅打包应用程序（及部分功能依赖） | 可执行文件 + 应用运行时的整个系统环境 |
| 跨平台 | 依赖特定的操作系统环境、发行版 | 具有非常好的跨平台便携性和兼容性 |
| 运行依赖 | 需在支持相应包管理的系统下安装，常需处理依赖 | 装有 docker 的机器拉下来就能用 |
| 磁盘空间 | 各包独立存储 | 分层结构，相同层只存一份可共享（后文讲） |

镜像让开发者在一个系统上开发（如 Ubuntu），打包成镜像后去另一个系统运行（如 CentOS），完全不用考虑环境依赖，是一种更高级的应用打包方式。

> 镜像屏蔽了操作系统，但硬件体系差异仍然存在，32/64 位、arm/x86 必须用不同的镜像。

## 容器化的应用

`docker pull busybox` 获取一个打包了 busybox 应用及其完整运行环境的镜像；`docker run busybox echo hello world` 提取镜像里的信息，运用 namespace、cgroup、chroot 技术创建隔离环境，再运行 busybox 的 echo 命令输出 "hello world"。

由于这两步基于标准 Linux 系统调用和只读镜像文件，所以无论在哪种操作系统、用哪种容器实现技术，都会得到完全一致的结果。

推而广之，任何应用都能用这种形式打包再分发后运行，这就是"一次编写，到处运行（Build once, Run anywhere）"的境界。

> 所谓"容器化的应用"（或"应用的容器化"），就是应用程序不再直接和操作系统打交道，而是封装成镜像，再交给容器环境去运行。

> 镜像就是静态的应用容器，容器就是动态的应用镜像，两者互相依存、互相转化、密不可分。

## Docker 的核心处理对象

在 Docker 官方架构图中，Docker 的核心处理对象就是镜像（image）和容器（container）。镜像是容器运行的根本，先有镜像才有容器。
