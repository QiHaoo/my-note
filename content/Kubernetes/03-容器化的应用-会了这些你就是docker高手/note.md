## 什么是容器化的应用

镜像（image）是容器的静态形式：把应用程序及其整个运行环境（文件系统、依赖库、环境变量、启动参数等）连同操作系统根目录一起打包，固化成一个只读的静态文件。用容器技术运行镜像，就形成了动态的容器，应用因此具备跨平台便携性，实现"一次编写，到处运行（Build once, Run anywhere）"。

所谓"容器化的应用"，就是应用程序不再直接和操作系统打交道，而是封装成镜像，再交给容器环境运行。

| 比喻 | 含义 |
|------|------|
| 镜像 = 样板间 | 把运行进程所需的文件系统、依赖库、环境变量、启动参数等打包整合到一起 |
| 容器 = 小板房 | 操作系统按"样板间"快速重建容器，应用看到一致的运行环境 |

与 tar / rpm / deb 安装包相比，镜像不仅包含可执行文件，还包含应用运行时的整个系统环境，因此具备更好的跨平台兼容性：

| 维度 | tar/rpm/deb 安装包 | 容器镜像 |
|------|-------------------|----------|
| 内容 | 打包应用程序 | 可执行文件 + 应用运行时的整个系统环境 |
| 跨平台 | 强依赖特定操作系统 | 可在 Ubuntu 开发、打包后到 CentOS 运行，无需考虑环境依赖 |

> 镜像就是静态的应用容器，容器就是动态的应用镜像，两者互相依存、互相转化、密不可分。

## 镜像的命名

镜像的完整名字由**名字**和**标签**两部分组成，中间用 `:` 连接：

- **名字**表明应用身份，如 busybox、alpine、nginx、redis
- **标签（tag）**区分不同版本，任意字符串都可以，如 `3.15`、`jamy`、`1.21-alpine`
- 特殊标签 `latest` 是默认标签：只给名字不附标签时自动使用

## 常用镜像操作

| 命令 | 作用 | 示例 |
|------|------|------|
| `docker pull <镜像>` | 从远端仓库拉取镜像 | `docker pull alpine:3.15`、`docker pull redis`(默认 latest) |
| `docker images` | 列出本地镜像，显示 REPOSITORY、TAG、IMAGE ID、CREATED、SIZE | `docker images` |
| `docker rmi <镜像>` | 删除不再使用的镜像 | `docker rmi redis`、`docker rmi d4c`(用 IMAGE ID 前三位) |

**IMAGE ID** 是镜像唯一标识（类似身份证号）。同一镜像可打不同标签，不同标签的 IMAGE ID 相同；它是十六进制且唯一，本地使用通常只需写前三位即可定位。

## 常用容器操作

`docker run` 是最复杂的容器操作命令，基本格式是"docker run 设置参数" + "镜像名或 ID" + "运行命令"：

| 参数 | 作用 |
|------|------|
| `-it` | 开启交互式 Shell，直接进入容器内部（`-i` 与 `-t` 组合） |
| `-d` | 让容器在后台运行（启动 Nginx、Redis 等服务器时很有用） |
| `--name` | 为容器起名，不给则 Docker 分配随机名 |
| `--rm` | 运行完毕自动清除容器，省去手工管理 |

每个容器都有唯一标识 CONTAINER ID，用法同 IMAGE ID。其他常用操作：

| 命令 | 作用 |
|------|------|
| `docker ps` | 查看当前运行的容器 |
| `docker ps -a` | 查看所有容器，含已停止的 |
| `docker exec -it <容器> sh` | 在运行中的容器里执行程序，最常用是用 `-it` 打开 Shell 进入容器 |
| `docker stop <容器>` | 强制停止运行中的容器 |
| `docker start <容器>` | 再次启动已停止的容器 |
| `docker rm <容器>` | 彻底删除容器（注意与 `docker rmi` 区别：无字母 i，只删容器不删镜像） |

`docker run` 用命令启动新容器，`docker exec` 在已有容器里执行命令（容器必须运行中）。容器被停止后用 `docker ps` 看不到，但并未销毁，需用 `docker rm` 删除。
