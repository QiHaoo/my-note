Docker 提供三种网络模式：null、host、bridge。处理网络通信是计算机系统里最棘手的工作之一，这里只从"宏观"层面讲个大概。

## 三种网络模式对比

| 模式 | 特点 | 通信效率 | 隔离性 | 启用参数 |
|------|------|----------|--------|----------|
| null | 没有网络，但允许其他网络插件自定义网络连接 | - | - | - |
| host | 直接使用宿主机网络，去掉容器网络隔离（其他隔离保留），所有容器共享宿主机 IP 和网卡 | 高（无中间层） | 差，运行太多容器容易端口冲突 | `--net=host` |
| bridge | 桥接模式，软件虚拟出网桥（docker0），容器和宿主机通过虚拟网卡接入，类似现实中的交换机/路由器 | 低一些（多了虚拟网桥和网卡） | 好 | `--net=bridge`（默认，一般不需显式指定） |

## host 模式

host 模式直接使用宿主机网络，相当于去掉容器的网络隔离（其他隔离依然保留），所有容器共享宿主机的 IP 地址和网卡。没有中间层，通信效率高，但缺少隔离，运行太多容器容易导致端口冲突。

```bash
docker run -d --rm --net=host nginx:alpine
```

验证效果：在本机和容器里分别执行 `ip addr` 查看网卡信息：

```bash
ip addr                      # 本机查看网卡
docker exec xxx ip addr      # 容器查看网卡
```

两个命令的输出完全一样，比如都是一个网卡 ens160，IP 地址 192.168.10.208，证明 Nginx 容器确实与本机共享了网络栈。

> host 一般用在集群的边界需要和集群外通信的场景，比如 ingress-nginx；host 性能好、隔离性差，适合小规模集群的简单拓扑结构。

## bridge 模式

bridge 即桥接模式，有点类似现实世界里的交换机、路由器，只不过由软件虚拟出来。容器和宿主机再通过虚拟网卡接入这个网桥（docker0），它们之间就可以正常收发网络数据包。和 host 模式相比，bridge 多了虚拟网桥和网卡，通信效率会低一些。

### 架构（文字描述）

bridge 模式下，宿主机通过虚拟网卡接入 docker0 虚拟网桥；Nginx 容器（虚拟网卡 eth0，IP 172.17.0.2）和 Redis 容器（虚拟网卡 eth0，IP 172.17.0.3）也各自通过虚拟网卡接入同一网桥 docker0，彼此即可收发网络数据包。

启动两个容器，没有特殊指定就会使用 bridge 模式：

```bash
docker run -d --rm nginx:alpine      # 默认使用桥接模式
docker run -d --rm redis             # 默认使用桥接模式
```

对比 host 模式，容器里的网卡设置与宿主机完全不同：eth0 是一个虚拟网卡，IP 地址是 B 类私有地址 172.17.0.2。

### 查看容器 IP

用 `docker inspect` 直接查看容器的 IP 地址：

```bash
docker inspect xxx | grep IPAddress
```

两个容器的 IP 地址分别是 172.17.0.2 和 172.17.0.3，宿主机则是 172.17.0.1，都在 172.17.0.0/16 这个 Docker 的默认网段，彼此之间就能用 IP 地址实现网络通信。

> 宿主机会由一个虚拟网卡接入 docker 的私有网段，docker 容器都在这个网段里。容器的 IP 是私有地址，要访问外网（如 baidu）肯定要经过宿主机网卡，即用宿主机的 IP 地址。

## 端口映射 -p

使用 host 或 bridge 模式，容器就有了 IP 地址，建立了与外部世界的网络连接，接下来要解决网络服务的端口号问题。

一台主机端口数量有限且多个服务不能冲突，但打包镜像通常都用默认端口，容器实际运行容易因端口被占用而无法启动。解决方法是加入"中间层"，由容器环境统一管理分配端口号，在本机端口和容器端口之间做"映射"操作——容器内部仍用自己的端口号，外界看到的是另一个端口号，避免冲突。

- 端口映射需使用 bridge 模式，并在 `docker run` 时用 `-p` 参数，形式和共享目录的 `-v` 类似，用 `:` 分隔本机端口和容器端口

启动两个 Nginx 容器，分别跑在 80 和 8080 端口：

```bash
docker run -d -p 80:80 --rm nginx:alpine
docker run -d -p 8080:80 --rm nginx:alpine
```

本机的 80 和 8080 端口分别映射到两个容器里的 80 端口，不会冲突。可用 `curl` 验证，`docker ps` 的"PORTS"栏可直观看到端口映射情况。

> 容器内的端口号是被隔离的，所以一样没问题，但外部映射的端口不能一样，否则就冲突了。bridge 模式下每个容器可以用任意端口，不用担心冲突，开发变简单了，维护成本也可以减少。
