镜像就是一个打包文件，里面包含应用程序和它运行所依赖的环境，例如文件系统、环境变量、配置参数等。

## 为什么需要分层

环境变量、配置参数比较简单，用 manifest 清单即可管理，真正麻烦的是文件系统。为保证容器运行环境一致，镜像必须把应用所在操作系统的根目录（rootfs）都包含进来。

这些文件不含系统内核（容器共享宿主机内核），但若每个镜像都重复打包，仍会导致大量冗余。例如一千个基于 Ubuntu 的镜像会重复一千次 Ubuntu 根目录，对磁盘和网络都是浪费。

解决思路：把重复部分抽取出来，只存一份 Ubuntu 根目录，让一千个镜像以某种方式共享这部分数据。这正是容器镜像的重大创新点——分层，术语叫 **Layer**。

## Layer 的结构与合并

容器镜像内部不是平坦结构，而是由许多镜像层组成：

- 每层都是只读不可修改的一组文件
- 相同的层可以在镜像之间共享
- 多个层像搭积木一样堆叠起来
- 再使用 **Union FS（联合文件系统）** 把它们合并，形成容器最终看到的文件系统

### 千层糕比喻

千层糕由很多层叠加，从最上面能看到每层镶嵌的葡萄干、核桃、杏仁等，每层糕相当于一个 Layer，干果相当于 Layer 里的文件。若某两层同一位置都有干果（文件同名），则只能看到上层文件，下层被屏蔽。

## 查看镜像分层：docker inspect

用 `docker inspect` 查看镜像分层信息，例如 nginx:alpine：

```bash
docker inspect nginx:alpine
```

分层信息在 "RootFS" 部分，nginx:alpine 镜像里一共有 6 个 Layer。

> `docker pull` 输出的 layer 是压缩数据的 sha256，`docker inspect` 里 layer 的 sha256 是解压后数据的 sha256，所以两者值不同。

## Layer 共享带来的收益

这也解释了 `docker pull`、`docker rmi` 等命令操作镜像时那些"奇怪"的输出——其实就是镜像里的各个 Layer：

- Docker 会检查重复层，本地已存在就不重复下载
- 层被其他镜像共享就不会删除
- 从而节约磁盘和网络成本

> 只有 RUN、COPY、ADD 会生成新的镜像层，其它指令只产生临时层，不影响构建大小。

## 容器运行时的可写层

镜像里的层只读不可修改，但容器运行时经常要写数据。解决方式：容器最上层是读写层，镜像所有层是只读层。容器启动后，Docker daemon 会在镜像上添加一个读写层。

- 读操作：直接访问镜像层或容器层的文件
- 写操作：利用写时复制（Copy-on-Write）技术；新建文件直接写入可写层
- 容器层是临时的，容器被删除时容器层也会删除，不影响镜像中的只读层
- 若需持久化数据，可使用 Docker 数据卷或绑定挂载

## 分层结构的好处

- 共享资源、节约空间，相同内容只需加载一份
- 可复用未改动的 layer，每次修改打包镜像只需重新构建被改动的部分
- 加速构建：Dockerfile 某一层没变就直接复用缓存，无需重新构建
- 减少网络传输，每次拉取推送只操作增量部分

> 层数太多会导致镜像加载效率低（最多约 127 层），所以要优化 Dockerfile。
